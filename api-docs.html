E-Commerce Partner API

API REST pública para integração de parceiros com os domínios de Cliente, Produto e Pedido da plataforma. Padrão arquitetural MVC, autenticação por API Key e versionamento via prefixo de URL.

Base URL: /api/v1 Node.js + Express Padrão MVC JSON · UTF-8 Versão 1.0.0
🔑
Autenticação
API Key — Header x-api-key

Todas as requisições para /api/v1/* devem incluir o header x-api-key com a chave fornecida pela equipe de parcerias. Chaves sem permissão recebem 401 Unauthorized. Em produção, o esquema é substituído por JWT (Bearer Token) com expiração configurável.

# Exemplo de requisição autenticada
curl -X GET https://api.ecommerce.com/api/v1/customers \
  -H "x-api-key: partner-key-001" \
  -H "Content-Type: application/json"
🏗
Arquitetura MVC
Model

Encapsula a lógica de negócio, validações e acesso a dados. Cada domínio tem seu próprio Model isolado.

models/Customer.js models/Product.js models/Order.js
View

Em APIs REST, a View é representada por respostas JSON padronizadas. Sem camada de template HTML.

→ JSON responses → Envelope padrão → Códigos HTTP semânticos
Controller

Orquestra a requisição HTTP: valida input, invoca o Model e formata a resposta. Sem lógica de negócio.

controllers/CustomerController.js controllers/ProductController.js controllers/OrderController.js 
── Fluxo de uma requisição ──────────────────────────────────

HTTP Request
  → Middleware (auth, logging, pagination)
    → Route  (src/routes/index.js)
      → Controller  (valida + orquestra)
        → Model  (lógica + dados)
          → Controller  (monta resposta)
            → HTTP Response (JSON)

Códigos de Status & Erros
200 Operação bem-sucedida (GET, PATCH, DELETE)
201 Recurso criado com sucesso (POST)
400 Requisição malformada
401 API Key ausente ou inválida
404 Recurso não encontrado
409 Conflito (e-mail/SKU duplicado)
422 Falha de validação de campos
500 Erro interno do servidor
